TAM Intelligence · pseudo-RAG over schema cards

How the system ingests data, and how it answers questions

Claude is the engine. Compact schema cards map the corpus; deterministic scripts pull exact rows. Every answer is grounded, cited to source rows, and dated.

Pipeline A

Ingestion — teaching the corpus a new file

The procedure is identical for every file and assumes nothing about its subject — only that it is a spreadsheet.

01

Profile — dump.py finds the real header row (not assumed row 1), infers types/roles, flags anomalies, keeps real source-row numbers.

02

Author cards — Claude writes one card per table from the file alone: what it is, what it answers, what it can't.

03

Date it — vintage inferred from evidence in the file; if none, marked unknown and flagged for the owner.

04

Normalize — entity names resolved to canonical form; new variants learned into the alias map.

05

Link + index — joins found by value overlap; the compact routing index is rebuilt.

06

Verify — each use-case is run through query.py; a card that can't answer its own questions gets enriched and retried.

Pipeline B

Querying — answering a question from the data

Route over cards, retrieve exact rows, compose a cited and dated answer — or say honestly what the corpus can't answer.

01

Route — Claude scans the small index and picks the tables whose use-cases match; it never reads every card.

02

Resolve — loads the chosen cards and normalizes company names in the question ("TRV" → the canonical account).

03

Retrieve — a query spec runs against the raw sheet, returning exact rows with their source-row citations.

04

Answer — grounded, dated, cited; if the data can't answer, it names the gap and offers the closest figure.

Time is a first-class citizen. Every card carries an as-of date, every answer leads with it, and stale sources trigger a refresh nudge — so a reader always knows they're seeing "Travelers as of Feb 2021," not today.

Packaging & sharing with the team

A skill file alone isn't enough — the cards, the alias map, the scripts, and the actual data files all have to travel together, because query.py reads the source sheets live.

What ships as one package

.claude/skills/tam-{ingest,ask,report} code/tam/*.py (engine) produced_data/cards/ + index.json aliases.csv input_data/ (the source spreadsheets)

Shared internal repo + Claude Code recommended

For anyone who can open a repo in Claude Code (web, desktop, or CLI).

  • The repo is the package — skills auto-discovered, scripts + cards + data all present, paths resolve with zero setup.
  • Access = repo permissions, which keeps confidential competitive data inside the org.
  • Versioned & updatable — re-run tam-ingest on new files, commit, everyone pulls.

Shared Project on claude.ai

For sales reps who live in claude.ai, not a repo.

  • A Team/Enterprise Project holds the data files + cards + scripts; the three skills are added as org Skills.
  • Reps just chat; Claude runs the bundled scripts via code execution.
  • Needs a one-time bundling step and a configurable data-root so paths resolve in the Project.
Confidentiality. The corpus is internal EXL competitive intelligence — distribute only within the org (private repo or Enterprise workspace). Never publish these skills or data publicly.
